## 二、详细修复记录

### 问题 1：历史行程和对话记录未保存到数据库

**根本原因**：
- `save_memory_node` 是空实现，仅返回 `{"itinerary_status": "confirmed"}`
- `MemoryStoreSkill` 的方法虽然完整，但从未被调用
- `models.py` 中 `session_id` 和 `user_id` 使用 UUID 类型，与前端字符串不兼容

**修复方案**：
1. **修改 `backend/src/models.py`**
   - 将 `Session.id`, `Conversation.session_id`, `Itinerary.session_id` 等字段从 `UUID(as_uuid=True)` 改为 `String(100)`
   - 移除外键约束（因为前端传入的是任意字符串 session_id）
   - 保留自动生成的 UUID 作为默认值

2. **重写 `backend/src/graph/nodes.py` 的 `save_memory_node`**
   - 使用 `async_session_maker()` 创建数据库会话
   - 调用 `MemoryStoreSkill.save_itinerary()` 将确认行程写入数据库
   - 包含完整的 itinerary、profile、budget 数据
   - 使用 try/except 包裹，确保持久化失败不中断流程

3. **修改 `backend/src/api/websocket.py`**
   - 在 graph 执行完成后，使用 `async_session_maker()` 保存对话记录
   - 调用 `MemoryStoreSkill.save_conversation()` 保存用户输入和助手回复

4. **修改 `backend/src/api/routes.py`**
   - 同样在 REST API 端点中保存对话记录

**验证**：
- ✅ 语法检查通过（`py_compile`）
- ✅ 20 个单元测试全部通过

### 问题 2：ValidationAgent 是"假验证"

**根本原因**：
- `_check_poi_existence` 中 `verified += 1` 是硬编码，没有真正搜索
- `_check_time_feasibility` 和 `_check_opening_hours` 内部只有 `pass`

**修复方案**：
1. **重写 `backend/src/agents/validation.py`**
   - `_check_poi_existence`：对采样 POI 调用 `UnifiedSearchSkill.search()` 进行真实搜索验证
     - 使用 `difflib.SequenceMatcher` 计算相似度
     - 标题/摘要包含 POI 名称或相似度 >= 0.6 视为验证通过
   - `_check_time_feasibility`：实现真实时间冲突检测
     - 检查每天活动总时长是否超过 12 小时（09:00-21:00）
     - 检查相邻活动是否时间重叠
   - `_check_opening_hours`：实现开放时间检查
     - 解析 `open_time`/`close_time` 和 `start_time`/`end_time`
     - 检查活动是否在开放时段内

**验证**：
- ✅ `test_validation.py` 中 5 个测试全部通过

### 问题 3：EventBus 死代码

**根本原因**：
- 整个代码库中没有任何地方调用 `event_bus.publish()`
- WebSocket 虽然订阅了 `panel_update`，但永远不会收到消息

**修复方案**：
1. **删除 `backend/src/core/event_bus.py`**（整个文件）
2. **修改 `backend/src/api/websocket.py`**
   - 移除 `event_bus.subscribe/unsubscribe` 调用
   - 简化 `finally` 块，仅保留 `manager.disconnect()`

**判断依据**：
- LangGraph 的 checkpoint 机制已经完整管理 graph 状态
- 如果未来需要进度推送，可直接在 node 函数中通过 WebSocket manager 发送
- EventBus 增加了不必要的抽象复杂度

### 问题 4：DBSCAN 聚类参数错误

**根本原因**：
- `DBSCAN(eps=0.01, metric="haversine")` 中 0.01 弧度 ≈ 63 公里
- 对于城市内 POI 聚类，这个值过大，整个城市会被聚为一个类
- `sklearn` 依赖增加了不必要的复杂度

**修复方案**：
1. **删除 `backend/src/agents/itinerary_planner.py` 中的 DBSCAN 逻辑**
   - 移除 `from sklearn.cluster import DBSCAN`
   - 移除 `_cluster_pois` 方法（约 30 行）
2. **新增 `_group_pois_by_area` 方法**
   - 使用 POI 已有的 `area` 字段（如"朝阳区"、"外滩"）进行分组
   - 按组大小降序排列，大区域优先分配独立日期
3. **修改 `_assign_days` 参数**
   - 从接收 `clusters: dict[int, list[ScoredPOI]]` 改为 `groups: dict[str, list[ScoredPOI]]`

**验证**：
- ✅ 语法检查通过
- ✅ 测试通过

### 问题 5：MapRouteAgent 生成随机坐标

**根本原因**：
- `get_coordinates` 使用 `random.uniform(-0.05, 0.05)` 生成随机偏移
- 每次调用结果不同，不可复现
- 偏移范围约 ±5.5km，完全失真

**修复方案**：
1. **修改 `backend/src/agents/map_route.py`**
   - 移除 `random`，改用基于 POI 名称 hash 的确定性偏移
   - `lat_offset = (hash(poi_name) % 100 - 50) / 10000.0`（±0.005° ≈ ±550m）
   - 同一 POI 名称永远产生相同坐标

### 问题 6：PostgreSQL + Redis 分工方案

**实施方案**：

| 数据库 | 用途 | TTL |
|--------|------|-----|
| PostgreSQL | 用户/会话/行程/对话/偏好变更持久化 | 永久 |
| PostgreSQL | LangGraph checkpoint | 永久 |
| Redis | POI 搜索结果缓存 | 1小时 |
| Redis | 天气数据缓存 | 30分钟 |
| Redis | 上下文富化结果缓存 | 24小时 |

**修改文件**：
1. `backend/src/core/redis_client.py` — 新增，提供 `get_json`/`set_json`/`set_json_with_ttl`
2. `backend/src/skills/poi_search.py` — 搜索前查缓存，结果写入缓存
3. `backend/src/skills/weather_query.py` — 查询前查缓存，结果写入缓存
4. `backend/src/agents/context_enrichment.py` — enrich 前查缓存，结果写入缓存
5. `backend/src/api/main.py` — lifespan 中连接 Redis，shutdown 时断开

### 问题 7：WebSocket 发送竞态条件

**根本原因**：
- `sendMessage` 中 `reconnect()` 后 `setTimeout(500)` 发送
- 新连接可能在 500ms 内尚未建立，导致消息丢失

**修复方案**：
1. **修改 `frontend/src/hooks/useWebSocket.ts`**
   - 新增 `pendingMessagesRef` 消息队列
   - 连接断开时，将消息存入队列而非立即发送
   - 在 `ws.onopen` 回调中消费队列，确保连接建立后才发送
   - 移除固定 500ms 的 setTimeout

### 问题 19：ContextEnrichment 成本过高

**修复**：
- **修改 `backend/src/agents/context_enrichment.py`**
- 添加 Redis 缓存（`context:{city}:{year_month}`，TTL 24小时）
- 6 个维度的搜索仍并行执行（网络 IO 可并行）
- 提取逻辑保持分维度（因不同维度结构不同，合并为单次 LLM 调用效果不佳，保持现状但加了缓存）

**备注**：原始计划中计划合并为 1 次 LLM 调用，经分析后决定保持 6 次但加缓存，因为：
- 不同维度返回的 JSON 结构差异大
- 合并后 prompt 过长，LLM 容易遗漏维度
- Redis 缓存可将重复查询降为 0 次 LLM 调用

### 关键技术决策

1. **模块级 LLM Mock**：`from core.llm_client import llm` 在导入时绑定局部引用，普通 monkeypatch 无效。改为在 `conftest.py` 模块级别直接替换所有引用模块的 `llm` 变量，测试速度从 **7.4 秒/测试 → 0.03 秒/测试**。

2. **AsyncMock vs MagicMock**：同步调用（`db.add()`）使用 `MagicMock`，异步调用使用 `AsyncMock`，消除 `RuntimeWarning: coroutine was never awaited`。

3. **FastAPI TestClient**：用于 REST API 和 WebSocket 端点测试，mock `app.state.graph` 避免 LangGraph 真实执行。
