### 2.2 架构亮点（面试加分项）

#### ✅ 亮点 1：LangGraph 声明式编排（技术敏锐度）

项目从 V2 的 "代码式 Orchestrator + EventBus" 演进为 V4 的 **LangGraph StateGraph**，体现了对 AI 应用框架发展趋势的敏锐度。

- **Fan-out/Fan-in 并行模式**：POI 搜索、天气查询、预算初始化三个节点并行执行；校验、路线优化、预算计算三个节点并行执行。LangGraph 自动处理等待汇聚，替代了手写的 `asyncio.gather()`
- **条件路由边**：基于 intent 的动态路由（`route_after_intent`），支持 7 种意图类型的分发
- **Checkpointer 持久化**：PostgreSQL-backed checkpointer 实现对话状态自动保存和恢复

**面试话术**：
> "我最初用 Python asyncio + EventBus 手写编排，但发现随着 Agent 数量增加，状态管理和循环终止变得复杂。后来引入 LangGraph，将编排从命令式改为声明式，代码量减少约 50%，并行逻辑和状态持久化由框架内置处理。"

#### ✅ 亮点 2：分层清晰的 Agent-Skill 架构（设计能力）

- **Agent 层**：负责业务逻辑和 LLM 交互（如 `IntentRecognitionAgent` 用结构化输出做意图分类）
- **Skill 层**：负责外部工具调用（搜索、爬虫、缓存、数据库）
- **严格分层**：Agent 不直接操作数据库/缓存，通过 Skill 间接调用

**面试话术**：
> "我设计了 Agent-Skill 两层架构，Agent 专注业务决策和 LLM 交互，Skill 专注工具执行。这样当 Tavily API 被限流时，我可以只改 `TavilySearchSkill`，而不影响上层的 `RealtimeQueryAgent`。"

#### ✅ 亮点 3：完整的可观测性系统（工程思维）

`ThoughtLogger` 是一个超出一般学生项目水平的可观测性模块：

- **并行安全**：使用 `contextvars` 隔离并发节点的日志上下文
- **实时推送**：WebSocket 推送运行状态（当前步骤、已完成步骤、耗时、token 用量）
- **双格式持久化**：JSON（结构化）+ Markdown（可读）
- **TTFT 追踪**：Time To First Token，衡量 LLM 响应延迟
- **Step 装饰器**：`@log_step("intent_node")` 自动包裹节点执行

**面试话术**：
> "LLM 应用是黑盒，调试很困难。我设计了一个并行安全的 ThoughtLogger，用 contextvars 隔离并发节点的日志上下文，通过 WebSocket 实时推送给前端，同时保存 JSON + Markdown 双格式日志。这让我能追踪每次用户请求经过了哪些节点、每个节点耗时多少、调用了几次 LLM、用了多少 token。"

#### ✅ 亮点 4：输入安全防护（安全意识）

`input_guard.py` 实现了：
- 注入模式检测（"Ignore previous instructions", "```system" 等）
- 输入长度限制（2000 字符）
- 控制字符过滤
- 边界包裹（`[用户输入开始]`/`[用户输入结束]`）

虽然防护级别不算生产级，但**有安全意识**这一点在大厂面试中是加分项。

### 2.3 架构缺陷（面试减分项 / 深挖风险）

#### ❌ 缺陷 1：LangGraph 使用深度不足（"用了框架但没用透"）

项目虽然引入了 LangGraph，但**只使用了最基础的功能**：

| LangGraph 高级特性 | 本项目使用情况 | 差距 |
|-------------------|--------------|------|
| Subgraph（子图嵌套） | ❌ 未使用 | 技术方案文档里画了子图，但代码中 `generate_subgraph` 没有作为独立子图编译嵌入，所有节点都在主图 |
| `interrupt` 人机交互机制 | ❌ 未使用 | 技术方案文档写了 interrupt，但代码中用的是传统的 `waiting_for_confirmation` 布尔标记 |
| `Command(resume=...)` 恢复执行 | ❌ 未使用 | 同样停留在文档阶段 |
| Streaming（`astream` 实时输出） | ❌ 未使用 | 代码用的是 `graph.ainvoke()` 等待全部完成才返回 |
| Human-in-the-loop | ⚠️ 伪实现 | 前端通过 intent 识别 + 状态标记模拟，非 LangGraph 原生中断 |

**面试官可能问**：
> "你提到用了 LangGraph，那 `interrupt` 和 `Command` 你是怎么用的？如果用户确认行程后想再修改，Graph 是怎么从 checkpoint 恢复的？"

**当前代码的回答**："我通过 `waiting_for_confirmation` 布尔标记和前端状态管理来模拟的..."（面试官内心：哦，没用 LangGraph 的高级特性，只是用了个 StateGraph 当路由表）

#### ❌ 缺陷 2："并行"是伪并行（单进程内 asyncio）

代码中所有节点都在同一个 Python 进程中通过 `asyncio` 并发执行。真正的生产级 AI 应用通常需要：
- 节点级水平扩展（不同节点可以部署在不同服务上）
- 异步队列（Celery / RQ / ARQ）处理耗时节点
- 分布式状态存储

**面试官可能问**：
> "如果 POI 搜索节点耗时 10 秒，planner 节点又需要 15 秒，整个 Graph 执行一次要 30 秒，用户等不及怎么办？"

**当前代码的瓶颈**：所有请求都在单进程的 WebSocket handler 中同步等待 Graph 完成，没有后台任务队列。

#### ❌ 缺陷 3：缓存策略过于简单

Redis 使用场景：
- POI 搜索：TTL 1 小时，key 为 `pois:{city}:{md5(keywords)}`
- 天气数据：TTL 30 分钟
- 上下文 enrichment：TTL 24 小时

问题：
- 没有缓存预热机制
- 没有缓存穿透/雪崩防护
- 缓存粒度粗（整个城市 POI 列表作为一个 key）
- 没有缓存命中率监控

#### ❌ 缺陷 4：数据库设计缺乏深度

- 没有外键约束（`models.py` 中移除了外键，因为前端传入字符串 session_id）
- 没有索引优化（除了主键外看不到其他索引）
- 没有分表/分区策略（对话记录表会快速增长）
- Alembic 迁移文件未在项目中看到
- 查询没有使用连接池优化

## 四、核心功能模块分析

### 4.1 意图识别模块（IntentRecognitionAgent）

**实现方式**：LLM 结构化输出（JSON mode）+ 后处理规则

**亮点**：
- 使用 `response_format={"type": "json_object"}` 强制结构化输出
- 自然语言日期解析（"下周一"、"下周"、"明天"、"5月1日"等）
- 偏好变更检测（对比新旧 profile）

**风险点**：
- 日期解析是**纯正则表达式**，没有使用 `dateparser` 等库，维护成本高
- 意图分类是**闭集**（7 种），没有 "其他/未知" 意图的兜底处理
- 置信度阈值固定为 0.7，没有动态调整机制

**面试深挖风险**：
> "如果用户说 '我不太满意这个行程'，你的系统怎么处理？"

当前代码：`_is_vague_modification` 函数用关键词匹配判断是否为模糊修改请求。这是**规则驱动**的，不是 LLM 判断的。如果用户说 "我觉得还可以更好"，系统可能无法识别。

### 4.2 行程规划模块（ItineraryPlannerAgent）

**实现方式**：LLM 直接规划（主路径）+ 6 层算法 fallback

**亮点**：
- **双路径设计**：优先 LLM 规划（质量高），失败时 fallback 到算法规划（稳定性高）
- **Prompt Engineering 深度**：系统 prompt 包含 9 条铁律（时间分配、路线效率、季节考虑、文化地标保留等）
- **POI 上下文丰富**：向 LLM 提供兴趣匹配标记、文化地标标记、高价值标记、区域信息、最佳时间等
- **城市 fallback 数据**：12 个中国主要城市的内置 POI 数据（含真实坐标和门票价格）

**风险点**：
- LLM 规划使用 `max_tokens=8192`，但没有流式输出，用户需要等待完整生成
- Fallback 算法的 2-opt 路由优化使用的是**伪坐标**（基于 POI 名称 hash 的确定性偏移），不是真实地理坐标
- 没有考虑**交通时间**（城市内景点间地铁/打车需要多久）

**面试深挖风险**：
> "你的路线优化是基于真实地图数据吗？"

当前回答："不是，因为真实地图 API 需要付费和 API key，我使用了基于 hash 的确定性偏移来模拟不同 POI 的坐标差异..."（面试官：哦，是模拟的）

### 4.3 校验模块（ValidationAgent）

**实现方式**：5 个维度并行校验（asyncio.gather）

| 校验维度 | 实现方式 | 真实度 |
|---------|---------|--------|
| 预算合规 | 代码计算 | ✅ 真实 |
| 时间可行性 | 代码解析时间字符串，检查冲突 | ✅ 真实 |
| POI 真实性 | 采样 30% POI，调用搜索验证 | ⚠️ 简化（只搜标题和摘要） |
| 开放时间 | 检查 open_time/close_time | ⚠️ 大多数 POI 没有开放时间数据 |
| 偏好覆盖度 | set 交集计算 | ✅ 真实 |

**面试深挖风险**：
> "如果 LLM 生成了一个不存在的景点，你的校验能发现吗？"

当前回答："我会随机采样 30% 的 POI，调用搜索验证标题和摘要中是否包含该景点名称..."（面试官：30% 采样意味着 70% 可能漏检；而且只是搜索验证，不是权威数据源验证）

### 4.4 实时信息查询模块（RealtimeQueryAgent）

**实现方式**：Tavily Search API + LLM 结构化提取

**亮点**：
- Tavily 搜索获取实时网页信息
- 多层提取策略：Answer 提取 → Snippet 提取 → Fallback 数据
- Redis 缓存减少重复查询

**风险点**：
- **强依赖 Tavily API**：如果 Tavily 服务不可用或限流，系统核心功能（POI 搜索）直接降级到内置 fallback 数据
- **没有自建数据源**：所有 POI 信息都来自搜索引擎，没有本地知识库或 RAG
- **天气数据未验证**：代码中没有看到真实天气 API 的调用，可能也是模拟或简化实现

## 六、面试扛问能力分维度评估

### 6.1 "请介绍一下你的项目" —— 表达能力 ✅ 扛得住

**建议话术结构**（2 分钟版本）：

> "这是一个 LLM 驱动的智能旅行规划 Agent。用户通过自然语言对话输入需求，比如 '我想去成都玩 4 天，预算 3000'，系统会：
> 
> 1. **意图识别**：用 LLM 做结构化意图分类，提取目的地、天数、预算等实体
> 2. **信息收集**：如果信息不全，主动追问；如果完整，进入规划流程
> 3. **并行查询**：同时搜索景点、查天气、初始化预算面板
> 4. **行程规划**：LLM 根据候选景点和用户偏好生成每日行程，有 9 条铁律约束（比如高票价景点必须给足时间、区域集中不绕路）
> 5. **并行校验**：验证预算、时间冲突、POI 真实性、偏好覆盖度
> 6. **方案生成**：生成自然语言方案，等待用户确认
> 
> 技术架构上，后端用 FastAPI + LangGraph 编排 10 个 Agent，前端用 Next.js + WebSocket 实时通信。我独立完成了从设计到实现的全流程。"

### 6.2 "为什么选择 LangGraph？不用行不行？" —— 技术判断力 ⚠️ 有风险

**扛问要点**：
- ✅ 能说出 LangGraph 解决的问题：状态管理、并行执行、对话持久化

# 1. Subgraph 作为独立编译单元
# 行程生成子图独立编译，可复用、可测试
generate_subgraph = build_generate_subgraph().compile(checkpointer=checkpointer)

# 在主图中作为节点嵌入
main_builder.add_node("generate_subgraph", generate_subgraph)

# 2. Interrupt 原生人机交互
# 方案生成后主动中断，等待用户输入
from langgraph.types import interrupt

async def proposal_node(state: ItineraryState) -> dict:
    proposal = await generate_proposal(state)
    # 抛出中断，状态自动保存到 checkpointer
    user_action = interrupt({
        "type": "itinerary_proposal",
        "proposal": proposal,
        "actions": ["confirm", "modify", "cancel"]
    })
    return {"user_action": user_action}

# 恢复执行
# graph.ainvoke(Command(resume="confirm"), config)

# 3. Streaming 实时输出
# async for event in graph.astream(state, config, stream_mode="updates"):
#     if "proposal_node" in event:
#         await websocket.send_json(event["proposal_node"])
```

**收益**：
- Subgraph 让核心流程可独立测试和版本管理
- Interrupt 让等待用户确认成为框架原生能力，代码更简洁
- Streaming 让用户看到实时进度

#### 实践 B：API 网关与接入层

**大厂标准做法**：统一接入层处理横切关注点。

```
当前：CORS 全开放，无限流，无认证

最佳实践：分层 middleware
1. 速率限制（防止 LLM 调用被刷爆）
2. 认证（JWT/API Key）
3. 请求 ID 追踪（全链路日志关联）
4. 超时控制（防止 LLM 慢请求拖垮服务）

```

      - ./nginx.conf:/etc/nginx/nginx.conf
```

### 5.3 具体改进方案

| 改进项 | 当前 | 目标 | 工作量 | 面试加分 |

### 5.4 面试话术升级

**改进前**：
> 后端用 FastAPI，异步处理请求。

**改进后**：
> 后端采用异步微服务架构：API 层（FastAPI）只负责接收请求和推送状态，真正的 LangGraph 执行交给 ARQ Worker 队列异步处理。Worker 可以水平扩展，多实例并发处理不同用户的请求。LLM 调用层有 Semaphore 并发控制，防止触发 OpenAI 限流。数据库做了分区表和读写分离，历史查询走从库。服务容器化部署，Nginx 做负载均衡，配合 healthcheck 自动摘除异常实例。

## 写在最后：如何选择改进项

**如果你只有 1 周时间**：
> 做 1（测试）+ 5（CI/CD）+ 写一份漂亮的 README。这三件事能让你的项目在简历筛选和面试初始阶段脱颖而出。

**如果你有 2 周时间**：
> 加上 2（RAG）+ 3（ARQ 队列）+ 4（Streaming）。这四项组合起来，你可以在面试中讲一个完整的"从同步到异步、从黑盒到可观测"的演进故事。

**如果你有 1 个月时间**：
> 完成 Phase 1 + Phase 2 的全部内容。这时候你的项目可以跟大厂内部项目对标，面试官会把你当"有工程经验的候选人"而不是"学生项目"。

**最重要的建议**：
> 不要试图把所有改进都做了。面试官更想听到的是"你为什么要做这个改进"、"改进前后对比是什么"、"遇到了什么困难怎么解决的"。所以选择 3-4 个改进项，做到深度足够、能讲出故事，比做 10 个浅层改进更有价值。
